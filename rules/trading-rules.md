## テクニカル指標

### 使用指標一覧

| 指標 | パラメータ | 用途 |
|------|-----------|------|
| RSI | 14期間 | 過熱感（売られすぎ/買われすぎ） |
| SMA | 20, 50期間 | トレンド方向 |
| EMA | 12, 26期間 | トレンドの勢い（MACD算出用） |
| MACD | 12, 26, 9 | トレンド転換・勢い |
| ボリンジャーバンド | 20期間, 2σ | ボラティリティ・過熱感 |
| ATR | 14期間 | ボラティリティ（SL/TP計算、プレフィルター用） |

### ボラティリティ判定基準

**相対ATR比率**（ATR / ATR_EMA(50)）で評価。銘柄・市場に依存しない適応的な判定：

| 出力値 | 説明 |
|--------|------|
| `atr_ratio` | 連続値スコア（1.0 = 平均的、>1 = 平均以上、<1 = 平均以下） |

後方互換用の離散ラベル（`volatility`フィールド）：

| レベル | 条件 | 意味 |
|--------|------|------|
| high | atr_ratio > 1.5 | 平均より50%以上高いボラ |
| medium | 0.7 ≤ atr_ratio ≤ 1.5 | 平均的なボラ |
| low | atr_ratio < 0.7 | 平均より30%以上低いボラ（プレフィルターでスキップ） |

**設計根拠:**
- 固定閾値（ATR% 0.5%/1.0%）は銘柄によって不適切（FX vs 仮想通貨 vs 債券で大きく異なる）
- EMA(50)を使用し、SMAより素早くレジームチェンジに追従
- 連続値スコアによりポジションサイズの滑らかな調整が可能

**データ要件:**
- ATR_EMA(50)の計算には最低64期間（ATR14期間 + EMA50期間）のOHLCデータが必要
- データ不足時はatr_ratio=1.0（平均的）にフォールバック

### トレンド判定

| トレンド | 条件 |
|----------|------|
| 上昇 | 価格 > SMA(20) AND SMA(20) > SMA(50) |
| 下降 | 価格 < SMA(20) AND SMA(20) < SMA(50) |
| 横ばい | 上記以外 |

## ルールベース判断

### カテゴリ分割方式

条件を「位置系」と「勢い系」に分類。
位置系（RSI or %B）は**どちらか1つ**満たせば発火、勢い系は**オプショナル**（あれば加点）。

### Buy候補条件

**発火条件: 位置系ANY（1/2）、勢い系はオプショナル**

| カテゴリ | 条件 | 閾値 | 必須 |
|----------|------|------|:----:|
| 位置系 | RSI が売られ気味 | RSI < RSI_BUY_THRESHOLD（動的） | どちらか1つ |
| 位置系 | ボリンジャー下限付近 | %B < 30 | どちらか1つ |
| 勢い系 | MACD上抜け + 勢い確認 | MACD > Signal AND (フレッシュクロス5本以内 OR hist3本連続増加) | オプション |

### Sell候補条件

**発火条件: 位置系ANY（1/2）、勢い系はオプショナル**

| カテゴリ | 条件 | 閾値 | 必須 |
|----------|------|------|:----:|
| 位置系 | RSI が買われ気味 | RSI > RSI_SELL_THRESHOLD（動的） | どちらか1つ |
| 位置系 | ボリンジャー上限付近 | %B > 70 | どちらか1つ |
| 勢い系 | MACD下抜け + 勢い確認 | MACD < Signal AND (フレッシュクロス5本以内 OR hist3本連続減少) | オプション |

### 勢い系条件の設計根拠

**複合条件: MACD > Signal AND (フレッシュクロス OR ヒストグラム増加)**

勢い系はMACDクロス済みを前提とし、追加で勢い確認を要求する。

**フレッシュクロス（N本以内、デフォルト5）:**
- クロス直後は勢いが強い傾向があり、エントリー精度が高い
- N本以上前のクロスは「勢いが継続しているか不明」のため除外
- `config/currencies/*.json` の `thresholds.fresh_cross_lookback` でカスタマイズ可能
- ボラの高い銘柄（XAUUSDなど）は3本、安定した銘柄（USDJPYなど）は7本など調整可能

**ヒストグラム3本連続変化:**
```
# 符号を問わず、変化方向だけを見る
# Buy例: [-0.5, -0.3, -0.1] → マイナス圏だが増加中 = 勢い回復中 ✓
# Sell例: [0.5, 0.3, 0.1] → プラス圏だが減少中 = 勢い減退中 ✓
```
- MACDは遅行指標のため、クロス時点ではすでにエントリーが遅い場合がある
- ヒストグラムの変化方向を見ることで、勢いの継続を確認できる

### Buy/Sell同時成立時の動作

RSI閾値（Buy < 40、Sell > 60）により通常はBuy/Sell条件が同時に成立することはないが、理論上のエッジケースとして両方の条件が全て満たされた場合は **Signal = "Wait"** となる（どちらも発火しない）。

```
if BUY_POSITION_COUNT == 2 AND BUY_MOMENTUM_COUNT == 1:
    if SELL_POSITION_COUNT == 2 AND SELL_MOMENTUM_COUNT == 1:
        → Wait（コンフリクト回避）
    else:
        → Buy

elif SELL_POSITION_COUNT == 2 AND SELL_MOMENTUM_COUNT == 1:
    → Sell

else:
    → Wait
```

**設計根拠:**
- Buy優先ではなく、コンフリクト時はどちらも発火させない保守的な設計
- 両方の条件が成立する状況は市場の方向性が不明確な可能性がある

### %B（ボリンジャーバンド位置指標）

%B = (価格 - BB下限) / (BB上限 - BB下限) × 100

| 値 | 意味 |
|----|------|
| 0 | BB下限上 |
| 50 | BB中央（SMA20） |
| 100 | BB上限上 |
| < 0 | BB下限を下回る |
| > 100 | BB上限を上回る |

**設計根拠:**
- 旧方式（BB下限 + バンド幅×30%）は高ボラ時にゾーンが広がりすぎる問題があった
- %Bはバンド幅で正規化されるため、ボラティリティに依存しない一貫した閾値になる
- %B < 30 / %B > 70 は旧30%閾値と意味的に整合

### RSI動的閾値（ボラティリティ連動）

高ボラティリティ環境ではRSI閾値を厳格化し、ノイズシグナルを抑制する。

| 条件 | RSI Buy閾値 | RSI Sell閾値 | 理由 |
|------|-------------|--------------|------|
| atr_ratio > 1.5（高ボラ） | 30 | 70 | トレンド明確、厳格な閾値で精度向上 |
| atr_ratio <= 1.5（通常） | 40 | 60 | 押し目/戻りを早めに拾う |

**設計根拠:**
- 上昇トレンド中はRSI 40でも「売られすぎ」と判断できる
- 高ボラ環境（トレンド明確時）は30/70の厳格閾値でも十分な機会がある
- 通常環境では40/60の緩い閾値で押し目/戻りを早めに検出

### MTFフィルター（マルチタイムフレーム確認）

条件を満たしても、**4h足の**上位トレンドに逆行するエントリーはブロックされる。
ただし、**横ばい（SMA差が0.1%以内）の場合は許容**される。

| 方向 | 許可条件（4h足） |
|------|------------------|
| Buy | 4h SMA(20) > 4h SMA(50)（上昇トレンド）**または横ばい** |
| Sell | 4h SMA(20) < 4h SMA(50)（下降トレンド）**または横ばい** |

**横ばい判定:**
- `SMA差% = (SMA20 - SMA50) / SMA50 × 100`
- `|SMA差%| <= 0.1%` の場合、横ばいとみなし通過を許可

**設計根拠:**
- 1h足のMACD条件（EMA12 vs EMA26）と同一時間軸のSMA20 vs SMA50は高相関で重複していた
- 本来のMTF（マルチタイムフレーム）の意図は「上位時間軸でトレンド方向を確認」
- 4h足を使用することで、1hの短期シグナルと4hの中期トレンドの整合性を確認できる
- 例: 1hで買いシグナル発生 → 4hが下降トレンド → ブロック（逆張りリスク回避）
- **横ばい許容**: 4hが明確なトレンドでない場合、1hのシグナルを尊重（過度なフィルタリング回避）

### ATRフィルター（レンジ相場抑制）

低ボラティリティ環境ではシグナルを抑制する。

| 条件 | 閾値 | 動作 |
|------|------|------|
| 低ボラ | atr_ratio < 0.7 | Signal = "Wait" に強制変更 |
| 極端高ボラ | atr_ratio > 3.0 | Signal = "Wait" に強制変更 |

**設計根拠:**
- RSI 40/60 の閾値はトレンド相場では有効だが、レンジ相場ではノイズシグナルが増える
- `atr_ratio < 0.7`（平均の30%以下）は値動きが小さいレンジ相場の可能性が高い
- レンジ相場でのエントリーは利益が出にくく、スプレッド負けしやすい
- `atr_ratio > 3.0`（平均の300%以上）は極端な高ボラティリティで、スプレッド拡大・スリッページリスクが高い
- 経済指標発表時や異常な相場変動時に該当し、安定したエントリーが困難

### BBスクイーズフィルター（ボラティリティ収縮検出）

ボリンジャーバンドが極端に収縮している状態（スクイーズ）ではシグナルを抑制する。

| 条件 | 閾値 | 動作 |
|------|------|------|
| スクイーズ | band_width_pct < 2.0% AND 0.7 ≤ atr_ratio < 1.0 | Signal = "Wait" に強制変更 |

**band_width_pct = (BB上限 - BB下限) / BB中央 × 100**

**設計根拠:**
- ATRフィルター（atr_ratio < 0.7）を通過しても、BBが極端に収縮している場合がある
- BBスクイーズは「静かな相場」であり、%B条件が満たされても方向性が不明確
- スクイーズ状態は通常、ブレイクアウト前の兆候であり、方向が確定するまで待つのが安全
- atr_ratio >= 1.0 の場合はATRが十分にあるためスクイーズとみなさない

### 判断フロー

```
[1h足分析]
条件一致数 >= 2 → 候補（Buy/Sell）
  └─ ATRフィルター（atr_ratio < 0.7 OR > 3.0） → Wait（異常ボラ抑制）
条件一致数 < 2  → Wait（見送り）

[MTFフィルター]（4h足で確認）
候補がBuy  → 4h SMA20 > SMA50 または横ばい（差0.1%以内）なら通過、それ以外はWait
候補がSell → 4h SMA20 < SMA50 または横ばい（差0.1%以内）なら通過、それ以外はWait
```

## プレフィルター（AI分析スキップ条件）

### 概要

AI分析はコストがかかるため、明らかにトレードチャンスがない場合は事前にスキップする。

### スキップ条件

| # | 条件 | 判定値 | 理由 |
|---|------|--------|------|
| 1 | シグナルなし | `BUY_CONDITIONS < 2 AND SELL_CONDITIONS < 2` | Buy/Sell どちらも2/3条件未満で方向性がない |
| 2 | 低ボラティリティ | `VOLATILITY == "low"` | 値動きが小さく利益が出にくい |

### 判定ロジック

```
if BUY_CONDITIONS < 2 AND SELL_CONDITIONS < 2:
    → AI分析スキップ（理由: 方向性なし）

elif VOLATILITY == "low":
    → AI分析スキップ（理由: 値動き不足）

else:
    → AI分析実行（どちらかが2/3以上）
```

### スキップ時の動作

| 項目 | 値 |
|------|-----|
| AI_DECISION | `not_order` |
| AI_CONFIDENCE | `0` |
| AI_SUMMARY | スキップ理由を表示 |
| FINAL_DECISION | `not_order` |

### 実装

`scripts/lib/prefilter.sh` の `check_prefilter()` 関数で判定。

## Stop Loss / Take Profit 設定

### 概要

エントリー時に自動でStop Loss（損切り）とTake Profit（利確）の関連注文を設定する。
ATR（Average True Range）を基準に動的に計算し、リスクリワード比 1:2 を維持する。

### 計算方式

| 項目 | 計算方法 | 例（USDJPY, ATR=0.50） |
|------|----------|------------------------|
| Stop Loss | エントリー価格 ± (ATR × 1.5) | Buy@150.00 → SL: 149.25 |
| Take Profit | エントリー価格 ± (SL幅 × 2.0) | TP: 151.50 |

### 方向による計算

**Buy の場合:**
```
SL価格 = エントリー価格 - (ATR × SL倍率)
TP価格 = エントリー価格 + (SL幅 × リスクリワード比)
```

**Sell の場合:**
```
SL価格 = エントリー価格 + (ATR × SL倍率)
TP価格 = エントリー価格 - (SL幅 × リスクリワード比)
```

### 設定パラメータ

| パラメータ | デフォルト値 | 説明 |
|-----------|-------------|------|
| `sl_tp.enabled` | `true` | SL/TP機能の有効/無効 |
| `sl_tp.stop_loss.mode` | `"atr"` | 計算モード（atr/pips/percentage） |
| `sl_tp.stop_loss.multiplier` | `1.5` | ATR倍率 |
| `sl_tp.take_profit.mode` | `"ratio"` | 計算モード（ratio/atr/pips/percentage） |
| `sl_tp.take_profit.value` | `2.0` | リスクリワード比 |

### 通貨設定例（config/currencies/*.json）

```json
{
  "symbol": "USDJPY",
  "sl_tp": {
    "enabled": true,
    "stop_loss": {
      "mode": "atr",
      "multiplier": 1.5
    },
    "take_profit": {
      "mode": "ratio",
      "value": 2.0
    }
  }
}
```

### AI による動的 SL/TP 調整

#### 設計思想

**「モデルの精度向上に成果が依存する」設計**

- AI にチャートデータと指標を渡し、最適な SL/TP を判断させる
- ルールを細かく指定せず、AI の分析力に委ねる
- モデルが賢くなれば、自然と良い SL/TP が提案されるようになる

#### AI レスポンス形式

```json
{
  "decision": {
    "action": "go",
    "direction": "Buy"
  },
  "sl_tp": {
    "stop_loss": 155.50,
    "take_profit": 157.20,
    "reasoning": "直近安値155.45の少し下にSL設定。BBアッパー157.30手前でTP。"
  }
}
```

#### AI への指示

```
エントリーする場合（action: go）、最適な Stop Loss と Take Profit の価格を提案してください。
チャートの形状、サポート/レジスタンス、ボラティリティなどを総合的に判断してください。
理由も簡潔に説明してください。
```

#### 処理フロー

```
1. AI が sl_tp.stop_loss と sl_tp.take_profit を提案
   ↓
2. 値が存在する場合 → AI 提案値を使用
   値がない/無効な場合 → 静的計算（ATR基準）にフォールバック
   ↓
3. 発注時に SL/TP を設定
```

#### フォールバック条件

以下の場合は静的計算（ATR × 1.5, リスクリワード 1:2）を使用：
- AI が `sl_tp` を返さない
- `stop_loss` または `take_profit` が null/undefined
- 値が現在価格に対して不合理（SL が TP より遠い等）

### 今後検討

- 日次/週次の損失上限
- 連続損失時の休止ルール
- トレーリングストップ対応
