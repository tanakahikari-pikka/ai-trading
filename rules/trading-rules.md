## テクニカル指標

### 使用指標一覧

| 指標 | パラメータ | 用途 |
|------|-----------|------|
| RSI | 14期間 | 過熱感（売られすぎ/買われすぎ） |
| SMA | 20, 50期間 | トレンド方向 |
| EMA | 12, 26期間 | トレンドの勢い（MACD算出用） |
| MACD | 12, 26, 9 | トレンド転換・勢い |
| ボリンジャーバンド | 20期間, 2σ | ボラティリティ・過熱感 |
| ATR | 14期間 | ボラティリティ（SL/TP計算、プレフィルター用） |

### ボラティリティ判定基準

**相対ATR比率**（ATR / ATR_EMA(50)）で評価。銘柄・市場に依存しない適応的な判定：

| 出力値 | 説明 |
|--------|------|
| `atr_ratio` | 連続値スコア（1.0 = 平均的、>1 = 平均以上、<1 = 平均以下） |

後方互換用の離散ラベル（`volatility`フィールド）：

| レベル | 条件 | 意味 |
|--------|------|------|
| high | atr_ratio > 1.5 | 平均より50%以上高いボラ |
| medium | 0.7 ≤ atr_ratio ≤ 1.5 | 平均的なボラ |
| low | atr_ratio < 0.7 | 平均より30%以上低いボラ（プレフィルターでスキップ） |

**設計根拠:**
- 固定閾値（ATR% 0.5%/1.0%）は銘柄によって不適切（FX vs 仮想通貨 vs 債券で大きく異なる）
- EMA(50)を使用し、SMAより素早くレジームチェンジに追従
- 連続値スコアによりポジションサイズの滑らかな調整が可能

**データ要件:**
- ATR_EMA(50)の計算には最低64期間（ATR14期間 + EMA50期間）のOHLCデータが必要
- データ不足時はatr_ratio=1.0（平均的）にフォールバック

### トレンド判定

| トレンド | 条件 |
|----------|------|
| 上昇 | 価格 > SMA(20) AND SMA(20) > SMA(50) |
| 下降 | 価格 < SMA(20) AND SMA(20) < SMA(50) |
| 横ばい | 上記以外 |

## ルールベース判断

### Buy候補条件（2つ以上で発火）

| # | 条件 | 閾値 |
|---|------|------|
| 1 | RSI が売られ気味 | RSI < RSI_BUY_THRESHOLD（動的） |
| 2 | 価格がSMA20下方（押し目検出） | SMA20 - ATR < 価格 <= SMA20 |
| 3 | 勢いあり | MACD > シグナルライン |
| 4 | ボリンジャー下限付近 | %B < 30 |

### Sell候補条件（2つ以上で発火）

| # | 条件 | 閾値 |
|---|------|------|
| 1 | RSI が買われ気味 | RSI > RSI_SELL_THRESHOLD（動的） |
| 2 | 価格がSMA20上方（戻り検出） | SMA20 < 価格 < SMA20 + ATR |
| 3 | 勢い弱い | MACD < シグナルライン |
| 4 | ボリンジャー上限付近 | %B > 70 |

### %B（ボリンジャーバンド位置指標）

%B = (価格 - BB下限) / (BB上限 - BB下限) × 100

| 値 | 意味 |
|----|------|
| 0 | BB下限上 |
| 50 | BB中央（SMA20） |
| 100 | BB上限上 |
| < 0 | BB下限を下回る |
| > 100 | BB上限を上回る |

**設計根拠:**
- 旧方式（BB下限 + バンド幅×30%）は高ボラ時にゾーンが広がりすぎる問題があった
- %Bはバンド幅で正規化されるため、ボラティリティに依存しない一貫した閾値になる
- %B < 30 / %B > 70 は旧30%閾値と意味的に整合

### RSI動的閾値（ボラティリティ連動）

高ボラティリティ環境ではRSI閾値を厳格化し、ノイズシグナルを抑制する。

| 条件 | RSI Buy閾値 | RSI Sell閾値 | 理由 |
|------|-------------|--------------|------|
| atr_ratio > 1.5（高ボラ） | 30 | 70 | トレンド明確、厳格な閾値で精度向上 |
| atr_ratio <= 1.5（通常） | 40 | 60 | 押し目/戻りを早めに拾う |

**設計根拠:**
- 上昇トレンド中はRSI 40でも「売られすぎ」と判断できる
- 高ボラ環境（トレンド明確時）は30/70の厳格閾値でも十分な機会がある
- 通常環境では40/60の緩い閾値で押し目/戻りを早めに検出

### 条件2「SMA20プロキシミティ」の設計根拠

**ATRベース動的幅を採用した理由：**
- 固定値（±1%）では銘柄・タイムフレームによって広すぎ/狭すぎる問題があった
- ATRベースにより、ボラティリティに応じてゾーンが自動調整される
- 高ボラ局面ではゾーンが広がり、低ボラ時には狭まる

**Buy/Sell方向性分離の理由：**
- 旧条件（±1%バンド）はBuy/Sellで同一範囲だったため、両方に+1カウントされる問題があった
- 新条件は排他的：Buy条件は「SMA20の下側」、Sell条件は「SMA20の上側」のみ
- これにより同点問題が構造的に解消される

**境界値の扱い：**
- Buy: SMA20 - ATR < 価格 **<=** SMA20（SMA20を含む）
- Sell: SMA20 **<** 価格 < SMA20 + ATR（SMA20を含まない）
- 価格がSMA20ちょうどの場合はBuy条件のみ成立（実運用上はほぼ発生しない）

**SMA20傾き（slope）の扱い：**
- ルール条件からは除外するが、AI分析用に計算は維持
- AIが「SMA20は上向きだが価格が接近中」のような文脈判断に使用可能

### MTFフィルター（マルチタイムフレーム確認）

条件を満たしても、**4h足の**上位トレンドに逆行するエントリーはブロックされる。

| 方向 | 許可条件（4h足） |
|------|------------------|
| Buy | 4h SMA(20) > 4h SMA(50)（上昇トレンド） |
| Sell | 4h SMA(20) < 4h SMA(50)（下降トレンド） |

**設計根拠:**
- 1h足のMACD条件（EMA12 vs EMA26）と同一時間軸のSMA20 vs SMA50は高相関で重複していた
- 本来のMTF（マルチタイムフレーム）の意図は「上位時間軸でトレンド方向を確認」
- 4h足を使用することで、1hの短期シグナルと4hの中期トレンドの整合性を確認できる
- 例: 1hで買いシグナル発生 → 4hが下降トレンド → ブロック（逆張りリスク回避）

### ATRフィルター（レンジ相場抑制）

低ボラティリティ環境ではシグナルを抑制する。

| 条件 | 閾値 | 動作 |
|------|------|------|
| 低ボラ | atr_ratio < 0.7 | Signal = "Wait" に強制変更 |

**設計根拠:**
- RSI 40/60 の閾値はトレンド相場では有効だが、レンジ相場ではノイズシグナルが増える
- `atr_ratio < 0.7`（平均の30%以下）は値動きが小さいレンジ相場の可能性が高い
- レンジ相場でのエントリーは利益が出にくく、スプレッド負けしやすい

### 判断フロー

```
[1h足分析]
条件一致数 >= 2 → 候補（Buy/Sell）
  └─ ATRフィルター（atr_ratio < 0.7） → Wait（レンジ相場抑制）
条件一致数 < 2  → Wait（見送り）

[MTFフィルター]（4h足で確認）
候補がBuy  → 4h SMA20 > SMA50 なら通過、それ以外はWait
候補がSell → 4h SMA20 < SMA50 なら通過、それ以外はWait
```

## プレフィルター（AI分析スキップ条件）

### 概要

AI分析はコストがかかるため、明らかにトレードチャンスがない場合は事前にスキップする。

### スキップ条件

| # | 条件 | 判定値 | 理由 |
|---|------|--------|------|
| 1 | シグナルなし | `BUY_CONDITIONS < 2 AND SELL_CONDITIONS < 2` | Buy/Sell どちらも2/4条件未満で方向性がない |
| 2 | 低ボラティリティ | `VOLATILITY == "low"` | 値動きが小さく利益が出にくい |

### 判定ロジック

```
if BUY_CONDITIONS < 2 AND SELL_CONDITIONS < 2:
    → AI分析スキップ（理由: 方向性なし）

elif VOLATILITY == "low":
    → AI分析スキップ（理由: 値動き不足）

else:
    → AI分析実行（どちらかが2/4以上）
```

### スキップ時の動作

| 項目 | 値 |
|------|-----|
| AI_DECISION | `not_order` |
| AI_CONFIDENCE | `0` |
| AI_SUMMARY | スキップ理由を表示 |
| FINAL_DECISION | `not_order` |

### 実装

`scripts/lib/prefilter.sh` の `check_prefilter()` 関数で判定。

## Stop Loss / Take Profit 設定

### 概要

エントリー時に自動でStop Loss（損切り）とTake Profit（利確）の関連注文を設定する。
ATR（Average True Range）を基準に動的に計算し、リスクリワード比 1:2 を維持する。

### 計算方式

| 項目 | 計算方法 | 例（USDJPY, ATR=0.50） |
|------|----------|------------------------|
| Stop Loss | エントリー価格 ± (ATR × 1.5) | Buy@150.00 → SL: 149.25 |
| Take Profit | エントリー価格 ± (SL幅 × 2.0) | TP: 151.50 |

### 方向による計算

**Buy の場合:**
```
SL価格 = エントリー価格 - (ATR × SL倍率)
TP価格 = エントリー価格 + (SL幅 × リスクリワード比)
```

**Sell の場合:**
```
SL価格 = エントリー価格 + (ATR × SL倍率)
TP価格 = エントリー価格 - (SL幅 × リスクリワード比)
```

### 設定パラメータ

| パラメータ | デフォルト値 | 説明 |
|-----------|-------------|------|
| `sl_tp.enabled` | `true` | SL/TP機能の有効/無効 |
| `sl_tp.stop_loss.mode` | `"atr"` | 計算モード（atr/pips/percentage） |
| `sl_tp.stop_loss.multiplier` | `1.5` | ATR倍率 |
| `sl_tp.take_profit.mode` | `"ratio"` | 計算モード（ratio/atr/pips/percentage） |
| `sl_tp.take_profit.value` | `2.0` | リスクリワード比 |

### 通貨設定例（config/currencies/*.json）

```json
{
  "symbol": "USDJPY",
  "sl_tp": {
    "enabled": true,
    "stop_loss": {
      "mode": "atr",
      "multiplier": 1.5
    },
    "take_profit": {
      "mode": "ratio",
      "value": 2.0
    }
  }
}
```

### AI による動的 SL/TP 調整

#### 設計思想

**「モデルの精度向上に成果が依存する」設計**

- AI にチャートデータと指標を渡し、最適な SL/TP を判断させる
- ルールを細かく指定せず、AI の分析力に委ねる
- モデルが賢くなれば、自然と良い SL/TP が提案されるようになる

#### AI レスポンス形式

```json
{
  "decision": {
    "action": "go",
    "direction": "Buy"
  },
  "sl_tp": {
    "stop_loss": 155.50,
    "take_profit": 157.20,
    "reasoning": "直近安値155.45の少し下にSL設定。BBアッパー157.30手前でTP。"
  }
}
```

#### AI への指示

```
エントリーする場合（action: go）、最適な Stop Loss と Take Profit の価格を提案してください。
チャートの形状、サポート/レジスタンス、ボラティリティなどを総合的に判断してください。
理由も簡潔に説明してください。
```

#### 処理フロー

```
1. AI が sl_tp.stop_loss と sl_tp.take_profit を提案
   ↓
2. 値が存在する場合 → AI 提案値を使用
   値がない/無効な場合 → 静的計算（ATR基準）にフォールバック
   ↓
3. 発注時に SL/TP を設定
```

#### フォールバック条件

以下の場合は静的計算（ATR × 1.5, リスクリワード 1:2）を使用：
- AI が `sl_tp` を返さない
- `stop_loss` または `take_profit` が null/undefined
- 値が現在価格に対して不合理（SL が TP より遠い等）
